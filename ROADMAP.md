# jp-gofaker Roadmap

## 目的

`jp-gofaker` を「日本語ダミーデータをすぐに生成できる小さなライブラリ」から、
「Go アプリケーションで継続利用できる信頼性の高い日本語フェイクデータ基盤」へ育てる。

## 現状整理

### 強み

- API が小さく、`Generator` から直感的に使える
- `WithSeed` により再現性のあるテストデータを作れる
- `Person` / `Address` / `Phone` / `Company` の基本カテゴリが揃っている
- 基本的なテストがあり、`go test ./...` は通る

### 主要課題

- データセットやサンプルコメントに文字化けがあり、パッケージの信頼性を損ねている
- README が最小限で、用途・制約・サンプルが不足している
- 生成できるデータ種類がまだ少なく、実運用のユースケースを十分にカバーできていない
- データの妥当性や分布品質を保証する仕組みが弱い
- リリース運用、互換性方針、コントリビューション導線が未整備

## 成長方針

1. まず「日本語データとして正しい」状態を作る
2. 次に「導入しやすく、壊れにくい」状態を作る
3. その後「現場で欲しいカテゴリ」を増やす
4. 最後に「継続的に育つ OSS」の運用基盤を作る

## ロードマップ

### Phase 0: 土台修復

目標: 最低限の品質不安を解消し、公開に耐える状態にする。

- データセットの文字コードを修正し、文字化けを解消する
- `README.md` を拡充し、使い方・生成例・seed の意味・非対応範囲を明記する
- `example_test.go` の出力例を実際の日本語として読める状態に整える
- 主要 API の GoDoc コメントを追加する
- テスト名・コメントの可読性を整える
- 公開リポジトリと `go.mod` の module path が一致していることを確認する

完了条件:

- 生成される代表値が人間に読める自然な日本語になっている
- README だけで初回利用できる
- `pkg.go.dev` 上で API の意図が追える

### Phase 1: 品質保証の強化

目標: ライブラリとして安心して組み込める状態を作る。

- フォーマット検証だけでなく、データ内容の整合性テストを増やす
- テーブル駆動テストで各 generator の期待仕様を整理する
- seed 固定時の回帰テストを追加する
- 異常系を潰すため、空データセットや不正設定に対する防御を入れる
- CI を導入し、`go test ./...` を自動実行する
- Go のサポートバージョン方針を決める
- OSS 公開前の最低限の品質ゲートとして、main ブランチで常時テストが通る状態を維持する

完了条件:

- PR ごとに自動テストが走る
- 主要 API の仕様変更をテストで検知できる
- 利用者が「再現性のあるテストデータ生成」を安心して使える

### Phase 2: 開発者体験の改善

目標: 使い始めやすく、拡張しやすい API にする。

- `README` にユースケース別サンプルを追加する
- `PersonName` や `AddressValue` と同様に、まとまりある値オブジェクトを増やす
- `Option` を拡張し、将来的な locale / format / strictness 追加に備える
- `Phone` や `Address` の文字列表現を用途別に選べるようにする
- ベンチマークを追加し、生成コストを把握する
- 破壊的変更を避けるため、API 設計の原則を明文化する

候補機能:

- `Address().Value()` に建物なしオプション
- `Person().Name()` に姓のみ / 名のみ / フルネーム表記揺れ対応
- `Phone()` にハイフンありなし切替

完了条件:

- README のサンプルだけで主要ユースケースを再現できる
- API を増やしても命名と構造に一貫性がある
- ベンチマーク値を継続監視できる

### Phase 3: データカテゴリの拡張

目標: 実務で必要な日本語ダミーデータを広くカバーする。

- メールアドレス
- ユーザー名 / ログイン ID
- 生年月日 / 年齢
- 性別表現の拡張
- 企業属性
- 銀行口座風データ
- マイページや管理画面向けのサンプルプロフィール

優先度の高い追加候補:

- `Internet`: email, domain, URL, username
- `Profile`: birthday, age, bio, avatar alt text
- `Job`: occupation, skill, employment type
- `Finance`: bank name, branch name, account-like number
- `Commerce`: product name, category, SKU-like code

方針:

- 実在情報に見えすぎない安全なデータを優先する
- 日本語として自然でも、誤用されにくい設計にする
- 値オブジェクトを返す API を増やし、単発文字列 API は補助に留める

完了条件:

- Web アプリの seed データやテストフィクスチャの主要用途を一通り賄える
- 新カテゴリ追加時の設計パターンが確立している

### Phase 4: OSS としての運用整備

目標: 外部コントリビュータが参加しやすく、継続リリースできる状態にする。

- `LICENSE` を追加し、利用条件を明示する
- GitHub などで公開リポジトリとして運用し、導入導線を明確にする
- `CONTRIBUTING.md` を追加する
- Issue / PR テンプレートを整える
- リリースノート運用を決める
- semver ベースのタグ運用を始める
- ライセンス表記とデータソース方針を明確にする
- データ追加時のレビュー観点を定義する
- 初回公開に向けて `v0.1.0` の公開チェックリストを作る
- `pkg.go.dev` での見え方を確認し、README・GoDoc・タグの整合性を取る

完了条件:

- 新規コントリビュータが追加手順を迷わない
- リリース判断が属人化しない
- データ品質と API 品質の基準が文書化されている
